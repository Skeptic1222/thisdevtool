Understanding XSS Attack Types

Reflected XSS

The malicious script is included in a request (URL, form) and reflected back in the immediate response. The victim must click a crafted link. Not stored on the server.

Stored XSS

The payload is saved on the server (database, comment, profile) and executed whenever any user loads the affected page. More dangerous — no crafted link needed.

DOM-Based XSS

The vulnerability is in client-side JavaScript. The payload modifies the DOM without ever hitting the server. Common in innerHTML, document.write, and eval sinks.

Encoding Methods Explained

  • HTML Entities — Converts characters like < > & " to their HTML entity equivalents (&lt; etc.). Useful for testing context-aware output encoding.
  • URL Encoding (percent encoding) — Converts characters to their %XX hexadecimal representation. Essential for payloads injected into URL parameters, query strings, or path segments.
  • Base64 — Encodes the payload as Base64. Used with eval(atob(...)) or data: URIs to bypass string-matching filters.
  • Unicode Escapes — Converts characters to \uXXXX notation. JavaScript interprets these natively, allowing script injection through Unicode-unaware filters.
  • Hex Escapes — Converts characters to \xXX hexadecimal notation, another form recognized by JavaScript engines.
  • Double Encoding — Applies URL encoding twice. Some vulnerable applications decode input twice (e.g., in a WAF and then in the application), allowing double-encoded payloads to slip through.

Common XSS Filter Bypass Techniques

Security researchers and penetration testers use encoding to test whether an application's input sanitization is complete. A well-implemented application should reject or neutralize all of these variants. If one encoding slips through, it indicates the application is only checking for specific patterns rather than performing proper context-aware output encoding. The gold standard for XSS prevention is to encode output at render time, not just validate input — because what constitutes "valid" depends entirely on the context where the data will be used.

When testing for XSS, always start with the simplest payload: <script>alert(1)</script>. If that fails, try event handlers like <img src=x onerror=alert(1)>. If those fail, try encoding variants. If all fail, the application is likely properly escaping output. Document every test and result for your penetration testing report.

Responsible Disclosure

If you discover a real XSS vulnerability during authorized testing, follow responsible disclosure practices: report the vulnerability privately to the organization's security team or via their bug bounty program (HackerOne, Bugcrowd, etc.), give them a reasonable remediation window (typically 90 days), and work collaboratively to verify the fix before public disclosure. Do not publish proof-of-concept exploits targeting real users without coordination. This protects both users and maintains the trust that makes bug bounty programs viable.

XSS Prevention Checklist for Developers

  • Use a modern framework (React, Angular, Vue) that auto-escapes template output
  • Never use innerHTML, document.write, or eval with user-supplied data
  • Implement a strict Content Security Policy (CSP) — Content-Security-Policy: default-src 'self'
  • Set HttpOnly and Secure flags on session cookies
  • Use the SameSite cookie attribute to mitigate CSRF alongside XSS
  • Validate input type/format on the server side — but rely on output encoding as the primary defense
  • Audit third-party scripts — any script on your page can steal user data

OWASP Context-Dependent Output Encoding

The OWASP (Open Web Application Security Project) guidelines emphasize that correct XSS prevention depends entirely on the output context where user data is rendered. Data inserted into an HTML element body requires HTML entity encoding. Data placed inside an HTML attribute requires attribute encoding (escaping quotes and special characters). Data injected into JavaScript blocks requires JavaScript encoding (Unicode escapes or hex escapes). Data used within CSS properties requires CSS encoding. Data placed in URL parameters requires percent-encoding. A single encoding function cannot safely handle all contexts, which is why this tool generates multiple encoding formats simultaneously. Developers must apply the correct encoding for each insertion point, and frameworks like React, Angular, and Vue handle this automatically in template expressions, which is one of the strongest arguments for using a modern frontend framework.

Real-World XSS Attack Scenarios

XSS vulnerabilities have been found in virtually every major web application at some point. In 2005, the Samy worm exploited a stored XSS vulnerability in MySpace to add over a million friends to its creator's profile within 20 hours. In 2018, a DOM-based XSS vulnerability in Google Search allowed attackers to execute arbitrary JavaScript via crafted search URLs. E-commerce sites are particularly vulnerable because they handle user-generated content (product reviews, forum posts, profile data) that is displayed to other users without adequate sanitization. Bug bounty programs on platforms like HackerOne and Bugcrowd consistently list XSS as one of the most frequently reported vulnerability categories.

Testing Methodology for Security Researchers

When conducting authorized penetration testing for XSS, follow a systematic approach. First, map all input vectors: form fields, URL parameters, HTTP headers (Referer, User-Agent), cookies, and any data reflected in the response. Second, test each input with a simple probe like <script>alert(1)</script> to establish a baseline. Third, if basic payloads are blocked, use encoding variants from this tool to test filter bypass techniques. Fourth, test context-specific payloads: event handlers (onerror, onload, onfocus) for attribute contexts, javascript: URIs for href attributes, and expression injection for CSS contexts. Document every test case, the encoding used, and the result. Use the CSP Generator to build a Content Security Policy that provides defense-in-depth, and test your headers with the HTTP Header Inspector. For encoding payloads in other formats, the Base64 Encoder and URL Encoder are also useful companion tools.

WAF Bypass and Defense-in-Depth

Web Application Firewalls (WAFs) attempt to detect and block XSS payloads by pattern matching on request parameters. However, encoding transformations can circumvent pattern-based detection. Double URL encoding, mixed-case payloads, Unicode normalization exploits, and null byte injection are all techniques that have historically bypassed WAF rules. This is precisely why output encoding at render time is the primary defense, not input filtering. A WAF should be considered an additional layer, not a replacement for proper output encoding. The principle of defense-in-depth means combining multiple layers: input validation (reject obviously malicious input), output encoding (neutralize any data at render time), Content Security Policy (restrict script execution), and HTTPOnly cookies (prevent session theft even if XSS occurs).
